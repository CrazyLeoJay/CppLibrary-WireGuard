# Cookie挑战

**Cookie挑战是被动触发的，只有在WireGuard服务端检测到CPU过载时才会启动，而不是握手流程中的固定步骤。**

在展开之前，需要明确一点：**在正常情况下，你完全不会接触到Cookie挑战**。标准的WireGuard握手就是“发起握手 -> 响应握手”两个报文，连接就建立了。

Cookie挑战的完整流程，本质上是服务端在压力下引入的一个**前置验证**，目的是在完成昂贵的DH计算前，逼客户端多走一个网络往返来证明诚意。

它的全流程可以拆解为三个阶段，核心角色是**服务端（Responder）**和**客户端（Initiator）**。

### 1. 触发阶段：服务端进入“低负载”模式

这是整个流程的起点。客户端发起的不是特殊的“Cookie请求”，而是标准的**握手初始化消息**。

- **发起者**：客户端按常规流程，向服务端发送握手初始化消息（`Handshake Initiation`）。
- **检测**：服务端的CPU正在忙于处理大量并发握手请求（例如遭遇DoS攻击）。此时它收到了这条新的初始化消息。
- **决策**：为了保护CPU资源，服务端的**Cookie检查器**决定不再直接响应该请求，而是进入“Under Load”状态。实现上，当积压的待处理握手数超过一定阈值（例如WireGuard-Go默认阈值约为128，Linux内核约为512）时就会触发。

### 2. 挑战与响应阶段：一个额外的网络往返

这是Cookie挑战执行的核心“一问一答”。

- **发起者**：服务端。
- **动作**：服务端会暂时搁置客户端发起的加密握手，转而生成并回复一个**Cookie回复消息**（`Cookie Reply`）。这个Cookie是通过**Cookie生成器**基于客户端`mac1`字段和一个仅服务端知道的密钥计算出来的。官网文档明确，当功能启用时，设备在此状态下**总是**发送cookie-reply报文。
- **接收**：客户端收到`Cookie Reply`。
- **响应者**：客户端。
- **动作**：客户端从报文中提取出Cookie，在重新构造的握手初始化消息中，将其填入`mac2`字段。注意，`mac2`本质上就是服务端下发的Cookie，用于证明客户端收到了上一步的回复。

### 3. 验证阶段：服务端放行

走完前面的“一问一答”后，服务端在收到重新发起的请求时才进行最终的校验。

- **发起者**：客户端（携带了包含`mac2`的新握手初始化消息）。
- **检测**：服务端再次收到该客户端的握手请求。
- **动作**：服务端的**Cookie检查器**提取出`mac2`，并验证其是否为自己刚才签发的、且未过期的Cookie。验证通过后，服务端才会消耗CPU资源进行后续的DH密钥交换，正常响应握手。
- **结果**：
  - **验证成功**：握手流程继续，隧道建立。
  - **验证失败**：服务端可能会再次回复Cookie，或直接丢弃报文。

**总结一下，这个机制的精巧之处在于**：它将DoS攻击的成本强行转嫁给了攻击者。攻击者为了消耗服务端的一次CPU计算，自己必须先完成一次完整的网络往返和Cookie计算，这使得无意义的泛洪攻击变得非常昂贵。

### 一些实现上的细节

- **Cookie的状态**：Cookie有过期时间。在FreeBSD的实现中可以看到，Cookie有一个对应的“出生日期”，过期后会被标记为无效。
- **密钥的安全性**：标准要求用于生成Cookie的密钥至少是32字节的强随机数，部分早期或实验性实现可能误用为32位，这会影响安全性。
- **对混淆层的挑战**：由于Cookie回复是明文且具有强特征的（`Under Load`状态会持续约1秒），它可以被网络审查者用作识别WireGuard流量的“指纹”。这也是为什么一些抗审查的WireGuard混淆实现，需要花很大力气去专门处理Cookie消息，甚至做出取舍。

由于你的工作涉及开发客户端，在处理Cookie挑战时，最核心的是要正确解析`Cookie Reply`消息，并将拿到的Cookie原样填入下一次`Handshake Initiation`的`mac2`字段中。如果想进一步了解这些消息的具体字段结构，可以参考WireGuard白皮书的第5.4.7节。
